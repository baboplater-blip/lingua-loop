# 운영 루프 가이드 (Operating the Evolution Loop)

> **"데이터로 끝없이 진화"를 실제로 돌리는 법.** 이 문서는 운영자가 무인 진화 폐루프를 어떤 주기로, 어떻게 실행·감시·롤백하는지 다룹니다.
> 전제: 자가호스팅(규칙 13, 외부 API 0) · append-only 이벤트(규칙 5) · **성과가 진실**(규칙 1) · **미검증 콘텐츠 노출 금지**(규칙 4).
> 모든 잡에는 안전판이 내장돼 있어, 잘못 눌러도 학습자 데이터나 서빙 품질을 해치지 않습니다.

관련 문서: 부팅·환경변수·API는 [`SELF_HOSTING.md`](SELF_HOSTING.md), 데이터 개방 원칙은 [`DATA_CARD.md`](DATA_CARD.md), 보안은 [`../SECURITY.md`](../SECURITY.md).

---

## 0. 폐루프 한눈에

```
  측정            분석              생성            게이트           발행           강등
POST /events → 격차·캘리브·리텐션 → 문항·읽기지문 → 규칙4 통과분만 → 서빙 편입 → 학습효과 나쁜 것 자동 강등
   ▲                                                                                    │
   └────────────────────────────  다시 학습자가 배우며 측정  ◀───────────────────────────┘
```

- 사람 개입 **0**으로 콘텐츠가 늘고(생성→게이트→발행), 나쁜 콘텐츠는 스스로 빠집니다(학습효과 강등).
- 잡은 **멱등**입니다 — 이미 채운 격차는 재생성하지 않으므로, 자주 돌려도(대개 no-op) 안전합니다.

---

## 1. 전제 — 무엇이 돌고 있어야 하나

1. **서버 구동** — `npm run serve` 로 학습 이벤트가 수집되고 있어야 합니다. 진화 잡은 신호를 이벤트에서 뽑습니다.
2. **같은 스토어를 가리키기** — 잡은 서버와 **동일한 스토어 환경변수**(`LL_DB` 또는 `LL_DATA_DIR`)를 써야 같은 이벤트를 봅니다. 다르면 잡이 빈 스토어를 읽어 "격차 없음"으로 조용히 끝납니다.
3. **데이터 축적 대기** — 초기(이벤트 적음)엔 잡이 "격차 없음/재적합 없음"으로 지나갑니다. **정상**입니다 — 진화는 학습이 쌓여야 신호가 생깁니다.

## 2. 진화 잡 — `evolve:publish` (언어별) · `evolve:all` (전 언어)

**하는 일**: 1사이클 = 콘텐츠 격차 분석 → 생성(문항 + 등급 읽기 지문, **게이트 통과분만**) → **자동 발행**(서빙 편입) → **FSRS 재적합**(복습 로그로 스케줄러 파라미터 재적합, **A/B 리텐션 가드레일 통과분만** append-only 영속 → `stateOf` 가 스케줄링에 적용·규칙 2) → **캘리브레이션**(수집 응답으로 난이도/변별도 데이터 추정·verified→calibrated 승격 영속·규칙 3) → 커뮤니티 기여를 학습효과로 재평가(강등) → 효능 스냅샷. 즉 한 잡이 자기개선 3축(**콘텐츠·스케줄·난이도**)을 함께 돌리고 측정한다.

> FSRS 재적합은 **성과(리텐션)를 개선할 때만 배포**된다(규칙 1). 예측오차만 줄고 A/B 리텐션이 나아지지 않으면 `미배포` — 임의 파라미터가 스케줄을 흔들지 않는다. 배포된 파라미터는 멱등 영속되어 모든 학습자 복습 스케줄에 반영된다.

```bash
# 한 언어
npm run evolve:publish                 # 기본 언어(LANG_PACK=en)
LANG_PACK=es npm run evolve:publish    # 언어 지정
LL_DB=/var/lib/lingualoop/ll.sqlite LANG_PACK=zh npm run evolve:publish

# 배포된 모든 타깃 언어를 순차로(운영 편의) — kc-seed.json 보유 팩을 자동 발견
npm run evolve:all                     # en·es·zh·ar·sw·ja·hi 전부
node scripts/evolve-all.mjs en es zh   # 특정 언어만
```

> `evolve:all` 은 `LL_DB`/`LL_DATA_DIR` 을 그대로 각 언어 잡에 물려줍니다. UI 전용 로케일(`ko` 등, `kc-seed.json` 없음)은 자동 제외됩니다.

**캘리브레이션만 따로 돌리기** — `evolve:publish` 에 이미 포함되지만, 응답이 크게 쌓였을 때 난이도만 자주 재추정하려면 독립 실행할 수 있습니다:

```bash
npm run calibrate                      # 기본 언어 — 수집 응답으로 IRT/ELO 난이도 추정(규칙 3)
LANG_PACK=es npm run calibrate         # 언어 지정
```

> 안전판: **이상 문항은 승격 제외**(응답 패턴이 비정상이면 SE 가 낮아도 verified 유지), 결과는 **append-only 영속**(멱등 — 값이 안 바뀌면 재기록 안 함)이라 서빙·Content Health 대시보드에 그대로 반영됩니다. 난이도는 임의값이 아니라 데이터로만 정해집니다(규칙 3).

### 출력 읽는 법

```
── 무인 진화 잡 (evolve → publish) ──
  언어                 en
  분석 이벤트           203         ← 스토어에서 읽은 학습 이벤트 수(0이면 스토어 변수 확인)
  콘텐츠 격차           4KC         ← verified 문항이 minItems(2) 미만인 KC 수
  생성                 문항 10 · 지문 7   ← 게이트 통과한 생성물(A1~B2 등급 읽기 포함)
  자동 발행             문항 10 · 지문 7 · 배제(중복) 0   ← 실제 서빙 편입 수 / 중복이라 건너뛴 수
  커뮤니티 재평가        승격 재검토 · 강등 0   ← 학습효과 나빠 강등된 기여 수
✅ 새 콘텐츠가 자동으로 서빙에 편입됐습니다.        ← 또는 ⏳(격차 없음=정상)
```

- **`✅ …편입`** vs **`⏳ 새로 발행할 격차가 없습니다(정상)`** — 후자는 실패가 아니라 "이미 다 채움"입니다.
- **멱등 실증**: 같은 언어를 연달아 두 번 돌리면 2회차는 `자동 발행 문항 0 · 지문 0`(⏳). 사이클 간 안정.

### 안전판 (왜 무인이 안전한가)

- 생성물은 코어 게이트(`checkItem`/`validateReading`, **규칙 4**)를 통과해야만 발행됩니다 — 미검증은 절대 노출되지 않습니다.
- 발행 뒤에도 학습효과가 나쁘면 **자동 강등**(규칙 1)돼 서빙에서 빠집니다. → 잘못된 콘텐츠가 오래 남지 않습니다.
- 격차 판정에 **이미 발행분·커뮤니티까지 포함**하므로 같은 격차를 재생성하지 않습니다(멱등).

## 3. 데이터셋 개방 잡 — `dataset:export`

**하는 일**: 개방 **동의** 이벤트만 → **k-익명성 재식별 레드팀** → 통과 시 JSONL + **데이터 카드** 동봉. 프라이버시 우선.

```bash
npm run dataset:export                          # datasets/open-dataset.jsonl, k=3(기본)
node scripts/export-dataset.mjs out.jsonl 5     # 출력 경로·k 지정(k↑ = 더 보수적)
```

### 출력 읽는 법 + 안전판

```
── 개방 데이터셋 내보내기 ──
  원본 이벤트           N
  개방 동의·학습신호     M (학습자 L)      ← 옵트인 안 했으면 0
  재식별 억제(singleton) X명 제외          ← 혼자뿐인 코호트는 뺌
  릴리스               학습자 · 이벤트
  k-익명성             최소 그룹 g / 목표 k=3 → ✅ 통과 / ❌ 실패
  라이선스             CC-BY-4.0
```

- **아무도 옵트인 안 함** → 빈 데이터셋, **파일 안 씀**(exit 0). 정상.
- **재식별 레드팀 실패**(최소 그룹 < k) → **파일 안 씀**(exit 1). k를 낮추거나 코호트를 키우세요.
- 성공 시 **`-card.md` 데이터 카드**가 자동 동봉됩니다 — **카드 없이 데이터 배포 금지**. 항상 함께 배포하세요.

## 4. 권장 캐던스

| 잡 | 권장 주기 | 근거 |
|---|---|---|
| `evolve:all`(또는 언어별 `evolve:publish`) | **일 1회**(저부하 시간) 또는 이벤트 N건마다 | 격차는 학습이 쌓여야 생김. 캘리브레이션 포함. 너무 자주 = 대개 no-op(무해). |
| `calibrate`(선택·독립 실행) | 응답이 많이 쌓인 뒤 **일 1회~주 1회** | 난이도 안정화용. `evolve` 에 이미 포함이라 필수는 아님. |
| `dataset:export` | **주 1회** 또는 릴리스 전 | 개방 데이터는 배치 성격. 매번 카드 재생성. |
| `npm run gate` | 배포·PR마다 | 완료 판정 단일 소스(문서 + 다크패턴 + 전체 테스트). |
| `node scripts/release.mjs` | 릴리스 전 | 준비도(게이트·시크릿 스캔·self-host 스모크·카드·라이선스). |

```cron

# 매일 03:00 — 전 언어 무인 진화(발견된 타깃 언어 전부)
```

## 6. 모니터링 — 무엇을 볼 것인가

### 효능 대시보드 (북스타 지표) — `npm run efficacy`

**참여도가 아니라 학습 성과**를 봅니다(규칙 1 성과가 진실). goal.md §3 북스타를 스토어 이벤트에서 산출:

```bash
npm run efficacy                 # 기본 언어(CLI 요약)
LANG_PACK=es npm run efficacy    # 언어별
# 웹 대시보드: 서버 구동 후 브라우저로  http://localhost:8787/ops.html  (언어 선택·바 시각화)
# 프로그램적으로: GET /efficacy?lang=<lang>  (JSON, 학습자 무관 집계)
```

| 지표 | 의미 | 지향(goal.md) |
|---|---|---|
| **TTM** 숙달까지 응답·경과 | 한 KC를 정답 2회(숙달)까지 걸린 노력/시간 | 코호트 대비 **단축** |
| **Retention** 전체·복습 정확도 | 회상 정확도, 특히 간격반복(이미 본 KC 재응답) | **상향** |
| **Coverage** 본/숙달 KC·학습자당 숙달 | 진척(얼마나 많은 능력요소를 숙달) | 상향 |
| **Gain Score** 사전→사후 θ·효과크기 d | 배치평가(사전) 대비 재평가(사후) 능력 상승과 관측 효과크기 | 상향·d 유의미 |
| **Content Health** 캘리브레이션 비율·격차 | 서빙 문항 중 IRT 캘리브레이션 비율·콘텐츠 없는 KC | 상향·격차↓ |

- **Gain Score 채우는 법**: 학습자가 **배치고사(레벨 탭)를 두 번 이상**(학습 후 다시) 보면 `assessment.item` θ가 사전·사후로 쌓여 상승과 효과크기 d가 산출됩니다. 재평가가 없으면 표본 0(`—`) — 진척을 다시 측정하도록 안내하세요.
- ⚠️ **인과 주의(규칙 17)**: Gain Score의 d는 **관측값**일 뿐 "도구가 가르쳤다"를 단정하지 않습니다. 인과 주장은 **사전등록 A/B 또는 사전-사후 통제군**이 필요합니다([`ab-experiment-framework`]). 항상 표본(n)과 함께 해석하세요.
- **쓰는 법**: **진화 사이클 전후로 비교**하세요. `evolve:all` 후 TTM이 줄고 Retention이 오르면 무인 진화가 실제로 학습을 개선한다는 증거(goal.md 자기개선 증명). 이벤트가 없으면 지표는 `—`(정상 — 사용이 쌓여야 채워짐).
- **추이(Loop Velocity)**: `evolve:publish`가 사이클마다 효능 **스냅샷**을 append-only로 기록합니다. `GET /efficacy/history?lang=`(또는 `POST /efficacy/snapshot`로 수동 기록)로 첫↔최신 델타(정확도↑·TTM↓·숙달 KC↑)를 봅니다. 웹 대시보드·`npm run efficacy`에도 추이가 표시됩니다.
- ⚠️ **세션수·스트릭은 노출하지 않습니다**(북스타 아님, rules.md C 다크패턴 금지).

### 사전등록 통제 실험 (관측 → 인과) — `npm run experiment`

Gain Score의 d는 *관측값*입니다(성숙·평균회귀·자기선택이 섞임). "이 개선이 **정말** 학습을 올렸다"를 주장하려면 **데이터를 보기 전에** 프로토콜을 고정하고(사전등록), 학습자를 통제군/실험군에 **무작위·결정적**으로 배정한 뒤, 두 집단의 Gain을 비교해야 합니다(규칙 17).

```bash
# 1) 데이터 수집 전에 사전등록 — 1차 결과는 학습성과(gainScore)로 고정(참여도 실험 금지·규칙 1)
#    intervention 을 주면 실험군에 실제 개입이 배선된다(선택). 지금은 practice_order(연습 순서)를 내장 지원.
curl -X POST localhost:8787/experiment -H 'content-type: application/json' \
  -d '{"experimentId":"exp-interleave-2026q3","hypothesis":"교차연습이 reading θ 상승을 키운다","treatmentShare":0.5,"minSamplePerArm":30,"intervention":{"kind":"practice_order"}}'

# 2) 앱이 학습자마다 배정 팔·연습 순서를 조회 — 배정은 결정적(재현 가능). 개입이 있으면 서빙이 자동 적용됨.
curl 'localhost:8787/experiment/assign?exp=exp-interleave-2026q3&learner=<ref>'
#    → {"variant":"treatment"|"control","practiceOrder":"interleaved"|"blocked"|null}

# 3) 충분히 쌓인 뒤 결과 — 집단 간 효과크기·95% CI·판정(운영자·실험 설계자용)
curl 'localhost:8787/experiment/result?exp=exp-cloze-2026q3'
npm run experiment exp-cloze-2026q3        # CLI 요약(한글)
```

- **판정 규율**: **사전 확정 표본(`minSamplePerArm`)을 두 팔 모두 채웠고** 평균차의 95% CI가 0을 배제할 때만 방향(`treatment_better`/`control_better`)을 주장합니다. 그 외에는 `no_difference`·`underpowered` — 소표본에서 섣부른 인과 결론을 막습니다.
- **가드레일(규칙 1)**: 1차 결과는 학습성과만. 참여도(세션·스트릭)를 결과로 두는 실험은 등록 자체가 거부됩니다. 실험군 개입이 리텐션을 떨어뜨리면(가드레일 위반) 채택하지 마세요.
- **사전등록 무효 감지**: 등록 시각이 학습 데이터보다 늦으면 결과에 `retroactive: true`가 붙습니다(사후 분석일 뿐 — 인과 주장 불가).
- **내장 개입 — 연습 순서(`practice_order`)**: 등록에 `intervention:{"kind":"practice_order"}`를 주면 서빙 계층(`/next`)이 배정 팔에 따라 실제로 다르게 제시합니다 — **실험군=인터리빙(교차연습)·통제군=블록연습**(SLA 근거 레버, 분량은 동일해 참여도 부풀리기 아님·규칙 1). `/experiment/assign`의 `practiceOrder`로 확인. **한 KC에 서빙 아이템이 2개 이상**일 때 순서 차이가 나타납니다(1개면 두 순서가 같음 = 정상).
- ⚠️ 새로운 개입(다른 생성기·스케줄 파라미터·콘텐츠)을 추가하려면 `core/efficacy-experiment.ts`의 `Intervention` 타입에 `kind`를 늘리고 서빙 계층에 배선하세요. 프레임워크는 사전등록·배정·측정을 제공하며, 개입은 이렇게 플러그인됩니다. 95% CI는 정규 근사이므로 소표본에서는 근사임에 유의하세요.

### 진화·데이터·건강

- **진화 로그**: `자동 발행` 줄. 며칠째 계속 큰 수면 격차가 잘 안 좁혀지는 것 → 시드/생성기 점검. 계속 `⏳`(0) → 정상(포화).
- **강등 수**: `커뮤니티 재평가 … 강등 N`. 급증 → 최근 기여 품질/어뷰즈 점검(`GET /contributions?rank=effect`).
- **건강 점검**: `npm run gate`(그린 유지) → 배포 전 `node scripts/release.mjs`(준비도).
- **데이터 개방**: `k-익명성 … ❌ 실패`가 반복되면 코호트가 작다는 신호 — k를 낮추기보다 개방 참여를 늘리는 게 원칙.

## 7. 롤백 · 안전 요약

| 상황 | 자동으로 어떻게 되나 | 운영자 조치 |
|---|---|---|
| 생성물이 저품질 | 게이트(규칙 4)가 발행을 **차단** | 없음(자동) |
| 발행됐는데 학습효과 나쁨 | 학습효과 **강등**(규칙 1)으로 서빙에서 제거 | 없음(자동) |
| 같은 잡 반복 실행 | **멱등** — 재발행 0 | 없음(자동) |
| 데이터셋 재식별 위험 | 레드팀 실패 시 **파일 미기록**(exit 1) | k·코호트 재검토 |
| 이벤트 유실 걱정 | **append-only** — 덮어쓰기 없음, 리플레이로 상태 복원 | 스토어 디렉터리/`.sqlite` 백업 |

> 이벤트는 절대 수정·삭제되지 않습니다(규칙 5·6). 잘못된 발행조차 새 이벤트로 강등될 뿐, 과거 기록은 그대로 남아 감사가 가능합니다.

### 리허설로 확인된 것 (2026-07-19)

무인 폐루프·롤백을 실제로 리허설해 다음을 확인했습니다(재현: `evolve:publish` 2회 + 강등 리허설):

- **무인 2사이클·멱등**: 격리 스토어에서 `evolve:publish` 1회차가 문항 13·지문 7 발행, 2회차는 격차가 없어 **재발행 0**(멱등). 각 사이클이 효능 스냅샷을 기록(추이 축적).
- **롤백(강등)**: 발행된 문항에 나쁜 학습효과(25응답 대부분 오답=too_hard)를 주입하니 `publishedBank`가 그 문항을 **자동 제외**(서빙에서 빠짐). 발행 이벤트 자체는 로그에 그대로 남아(append-only) 감사가 가능 — 강등은 *서빙 필터*일 뿐 이력 파괴가 아닙니다.

> 스케줄 실설치(§5 crontab/schtasks)는 운영자 환경에서 1회 등록하면 됩니다 — 위 리허설은 잡 자체가 무인으로 안전하게 도는지를 검증한 것입니다. 스케줄러 등록은 시스템 상태를 바꾸므로 운영자가 직접 수행하세요.
